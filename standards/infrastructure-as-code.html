Automated Infrastructure as Code Deployment Standard

Scope

In Scope

  • Terraform and OpenTofu module standards
  • GitOps workflows for infrastructure changes
  • Policy-as-code enforcement with OPA/Sentinel
  • Infrastructure drift detection and remediation
  • Module registry and shared module governance

Out of Scope

  • Application deployment pipelines (covered by CI/CD standard)
  • Container orchestration and Kubernetes configuration
  • Network infrastructure managed by network operations

Requirements (6)

Requirements use RFC 2119 severity levels.

REQ-001 MUST

All infrastructure changes MUST be deployed through version-controlled IaC pipelines.

Version-controlled IaC pipelines ensure auditability, reproducibility, and peer review of all infrastructure modifications, reducing the risk of configuration errors and unauthorized changes.

Verification: Audit CI/CD pipeline logs to confirm all production infrastructure changes originate from IaC pipeline runs; cross-reference with cloud provider activity logs.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3, Tier 4
REQ-002 MUST

Infrastructure modules MUST pass policy-as-code validation before apply.

Policy-as-code validation enforces security, compliance, and cost guardrails automatically before any infrastructure change is applied, shifting governance left in the deployment process.

Verification: Review pipeline configurations to confirm policy-as-code gates (OPA/Sentinel) are integrated and enforced prior to terraform apply stages.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3, Tier 4
REQ-003 MUST NOT

Manual infrastructure changes MUST NOT be made to production environments.

Manual changes introduce configuration drift, bypass peer review and policy checks, and create undocumented state that cannot be reliably reproduced or rolled back.

Verification: Enable cloud provider guardrails (e.g., AWS SCPs, Azure Policy) to restrict console-based modifications; run weekly drift detection scans to identify unauthorized changes.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3, Tier 4
REQ-004 SHOULD

Shared Terraform modules SHOULD be published to the internal module registry with semantic versioning.

A centralized module registry with semantic versioning promotes reuse, consistency, and controlled upgrades across teams while avoiding duplicated infrastructure logic.

Verification: Review module registry catalog for published modules; verify that each module uses semantic versioning and includes documentation and examples.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3, Tier 4
REQ-005 SHOULD

Infrastructure drift detection SHOULD run daily with automated alerting.

Daily drift detection identifies unauthorized or accidental changes promptly, enabling rapid remediation before drift compounds and causes operational issues.

Verification: Confirm drift detection jobs are scheduled daily in the CI/CD platform; validate that alerts are routed to the responsible platform team's notification channel.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3, Tier 4
REQ-006 MUST

State files MUST be stored in encrypted remote backends with state locking enabled.

Remote state with encryption protects sensitive infrastructure metadata at rest and in transit, while state locking prevents concurrent modifications that could corrupt infrastructure state.

Verification: Audit Terraform backend configurations to confirm remote storage (e.g., S3, GCS, Azure Blob) with encryption enabled and DynamoDB/equivalent locking mechanisms in place.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3, Tier 4

Guidelines

GUIDE-001

Module Structure: Organize Terraform modules using a consistent directory layout with separate directories for environments, modules, and configuration. Each module should have a README, input variables with descriptions, output definitions, and example usage. Pin provider versions and use dependency lock files.

GUIDE-002

State Management: Use separate state files per environment and per service boundary to limit blast radius. Enable versioning on state storage backends to allow rollback. Never store secrets in state files; use data sources or secret manager references instead. Implement state file access controls using IAM policies.

GUIDE-003

Secret Handling in IaC: Never hardcode secrets in Terraform code or variable files. Use dynamic secret retrieval from a secrets manager (e.g., HashiCorp Vault, AWS Secrets Manager) via data sources. Mark sensitive variables with the sensitive flag to prevent accidental exposure in logs and plan output.

Definitions

TermDefinition
GitOps An operational framework that applies DevOps best practices for infrastructure automation by using Git repositories as the single source of truth for declarative infrastructure and application configurations, with automated reconciliation ensuring the live environment matches the desired state.
Policy-as-Code The practice of expressing organizational policies, security rules, and compliance requirements as machine-readable code that can be version-controlled, tested, and automatically enforced within CI/CD pipelines and runtime environments.
Infrastructure Drift The divergence between the actual state of deployed infrastructure and the desired state defined in infrastructure-as-code configurations, typically caused by manual changes, failed deployments, or external modifications outside the IaC workflow.

Frequently Asked Questions

What should teams do when an emergency hotfix requires an immediate infrastructure change that cannot wait for the standard IaC pipeline?
Emergency changes should follow the documented break-glass procedure which allows temporary manual access with enhanced logging. All emergency changes must be retroactively codified in IaC within 24 hours, and a post-incident review must document the change and update runbooks to prevent recurrence.
How do we handle infrastructure that was provisioned before this standard was enacted?
Existing infrastructure should be imported into Terraform state using terraform import or equivalent tooling. Teams should prioritize importing Tier 1 and Tier 2 workloads within six months and complete all tiers within twelve months. The Platform Engineering team provides import tooling and migration guides.
