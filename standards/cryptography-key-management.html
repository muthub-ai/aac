Application Cryptography and Key Management Standard

STD-SEC-001 — Security • Version 2.0 • Approved Nov 1, 2024

APPROVED Standard v2.0 Security

Scope

In Scope

  • Symmetric and asymmetric encryption for data at rest and in transit
  • TLS certificate lifecycle management
  • Secrets management for application credentials and API keys
  • Hardware Security Module (HSM) integration
  • Key rotation policies and crypto-agility preparation

Out of Scope

  • Physical security of data centers
  • Network-level encryption managed by infrastructure teams
  • Blockchain or distributed ledger cryptographic protocols

Requirements (6)

Requirements use RFC 2119 severity levels.

REQ-001 MUST

All external-facing and internal service-to-service communications MUST use TLS 1.3 or higher for data in transit.

TLS 1.3 eliminates known vulnerabilities in prior versions including downgrade attacks and weak cipher suites, providing stronger confidentiality and integrity guarantees.

Verification: Automated TLS version scan across all endpoints using network security scanner integrated into CI/CD pipeline
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3
REQ-002 MUST

Cryptographic keys MUST be rotated according to defined schedules: symmetric keys every 90 days, asymmetric keys every 365 days.

Regular key rotation limits the window of exposure if a key is compromised and satisfies regulatory requirements for cryptographic hygiene.

Verification: Key management service audit report showing rotation compliance rates and alerting on overdue rotations
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3
REQ-003 MUST

Applications MUST use only approved cryptographic algorithms: AES-256 for symmetric encryption, RSA-2048 or higher and ECDSA P-256 or higher for asymmetric operations.

Using approved algorithms ensures adequate security strength and interoperability while avoiding deprecated or weak ciphers that may be exploited.

Verification: Static code analysis rules detecting usage of non-approved algorithms and cipher suites
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3
REQ-004 MUST

Tier 1 applications MUST store cryptographic keys in Hardware Security Modules (HSMs) or cloud-native HSM-backed key management services.

HSMs provide tamper-resistant hardware protection for key material, meeting the highest security requirements for critical business applications and regulatory compliance.

Verification: Architecture review checklist confirming HSM integration for all Tier 1 application key storage
Platforms: AWS, Azure, GCP
Tiers: Tier 1
REQ-005 MUST NOT

Applications MUST NOT store secrets, credentials, or cryptographic keys in source code, configuration files, or environment variables in plain text.

Hardcoded secrets are a leading cause of security breaches; secrets in code repositories persist in version history even after removal and are easily discovered by attackers.

Verification: Pre-commit hooks and CI/CD secret scanning tools detecting hardcoded credentials and keys in repositories
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3
REQ-006 SHOULD

Application architectures SHOULD incorporate crypto-agility patterns to enable rapid migration to post-quantum cryptographic algorithms when standards are finalized.

Quantum computing advances threaten current asymmetric algorithms; crypto-agile architectures enable faster migration to quantum-resistant algorithms without major refactoring.

Verification: Architecture review assessment of cryptographic abstraction layers and algorithm configurability
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2

Guidelines

GL-001

Use well-established and vetted cryptographic libraries (e.g., OpenSSL, BoringSSL, libsodium) rather than implementing custom cryptographic functions

GL-002

Automate secret rotation using native integration between secret management services and application deployment pipelines

GL-003

Implement automated certificate lifecycle management using ACME protocol to prevent certificate expiration outages

Definitions

TermDefinition
Key Rotation The process of retiring an active cryptographic key and replacing it with a new key, limiting the amount of data encrypted under a single key and reducing the impact of key compromise.
Crypto-Agility The ability of an information system to rapidly switch between cryptographic algorithms, key sizes, and protocols without significant changes to the system architecture or application code.
Hardware Security Module (HSM) A dedicated, tamper-resistant hardware device that provides secure generation, storage, and management of cryptographic keys and performs cryptographic operations within a protected boundary.

Frequently Asked Questions

Can we use AWS Secrets Manager instead of HashiCorp Vault for secrets management?
Yes. AWS Secrets Manager is an approved alternative for workloads running on AWS. The key requirement is that secrets are stored in an approved secrets management service rather than in code or plain-text configuration files.
When will post-quantum cryptographic algorithms become mandatory?
Post-quantum algorithms are currently a SHOULD recommendation while NIST finalizes standards. Once NIST publishes final post-quantum standards and our approved algorithm list is updated, a transition timeline will be communicated with a target migration window of 18-24 months.
