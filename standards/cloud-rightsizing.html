Cloud Workload Resource Rightsizing Standard

Scope

In Scope

  • Compute instance rightsizing recommendations and enforcement
  • Reserved instance and savings plan procurement guidelines
  • Spot/preemptible instance usage for fault-tolerant workloads
  • Cost allocation tagging and chargeback models
  • Idle resource detection and automated cleanup

Out of Scope

  • Application-level performance optimization
  • Software licensing cost management
  • Data transfer cost optimization across regions

Requirements (5)

Requirements use RFC 2119 severity levels.

REQ-001 MUST

All cloud resources MUST have cost allocation tags matching the enterprise tagging taxonomy.

Consistent cost allocation tagging is the foundation of cloud financial management, enabling accurate chargeback, showback, and cost attribution to business units, projects, and environments.

Verification: Run automated tag compliance scans using cloud-native tools (e.g., AWS Config Rules, Azure Policy) and report the percentage of resources with compliant tags on a weekly basis.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3, Tier 4
REQ-002 SHOULD

Rightsizing recommendations SHOULD be reviewed and actioned within 30 days of generation.

Timely review of rightsizing recommendations ensures cost savings opportunities are captured before cloud spending compounds, and keeps resource utilization aligned with actual workload demands.

Verification: Track the age of open rightsizing recommendations in the FinOps dashboard and report on the percentage actioned within the 30-day SLA each month.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3, Tier 4
REQ-003 MUST

Idle resources with no traffic for 14 days MUST be flagged for decommission review.

Idle resources represent direct waste in cloud spending. Automated detection and flagging ensures that unused resources are identified promptly and either justified or decommissioned.

Verification: Configure automated idle resource detection using cloud monitoring tools; verify that flagged resources are reviewed within 7 days and either decommissioned or documented with a business justification.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2, Tier 3, Tier 4
REQ-004 SHOULD

Teams SHOULD leverage spot instances for batch processing and fault-tolerant workloads.

Spot and preemptible instances offer 60-90% cost savings over on-demand pricing for workloads that can tolerate interruption, significantly reducing compute costs for batch, CI/CD, and data processing jobs.

Verification: Review compute spend reports to measure spot instance adoption rates for eligible workload categories; target a minimum of 50% spot usage for identified fault-tolerant workloads.
Platforms: AWS, Azure, GCP
Tiers: Tier 2, Tier 3, Tier 4
REQ-005 MUST

Reserved instance coverage MUST target minimum 70% of steady-state compute workloads.

Reserved instances and savings plans provide 30-60% cost savings for predictable workloads. Maintaining at least 70% coverage for steady-state compute ensures the organization captures significant savings on its baseline infrastructure spend.

Verification: Monitor reserved instance utilization and coverage metrics through cloud provider cost management tools; report coverage percentage monthly to FinOps steering committee.
Platforms: AWS, Azure, GCP
Tiers: Tier 1, Tier 2

Guidelines

GUIDE-001

Tagging Strategy: Implement a minimum set of mandatory tags for all cloud resources including cost-center, environment, application-id, owner-email, and project-code. Use tag enforcement policies at the account or subscription level to prevent resource creation without required tags. Automate tag remediation for non-compliant resources discovered during audits.

GUIDE-002

Spot Instance Patterns: Design fault-tolerant workloads to leverage spot instances using diversified instance type pools, capacity-optimized allocation strategies, and graceful interruption handling. Use spot fleet or managed instance groups with automatic fallback to on-demand capacity. Implement checkpointing for long-running batch jobs to minimize reprocessing on interruption.

GUIDE-003

Savings Plans and Reservations: Analyze 90 days of historical usage data before purchasing reserved instances or savings plans. Start with compute savings plans for flexibility, then layer in specific reserved instances for stable workloads. Stagger reservation purchases quarterly to avoid lock-in risks and allow adjustment as workload patterns evolve.

Definitions

TermDefinition
Rightsizing The process of analyzing cloud resource utilization metrics and adjusting instance types, storage tiers, or configurations to match actual workload requirements, eliminating over-provisioning while maintaining performance and availability targets.
Reserved Instance A billing discount mechanism offered by cloud providers where customers commit to a specific instance type and region for a one- or three-year term in exchange for significantly reduced hourly rates compared to on-demand pricing.
FinOps An evolving cloud financial management discipline and cultural practice that enables organizations to get maximum business value by helping engineering, finance, and business teams collaborate on data-driven spending decisions around cloud infrastructure and services.

Frequently Asked Questions

How do we handle rightsizing recommendations that conflict with application performance requirements?
Rightsizing recommendations should be validated against application-specific performance baselines before implementation. If a recommendation would degrade performance below acceptable thresholds, teams should document the exception with supporting metrics in the FinOps platform and re-evaluate during the next quarterly review cycle.
Should reserved instances be purchased centrally or by individual teams?
Reserved instances and savings plans should be purchased centrally by the FinOps team to maximize flexibility and coverage across the organization. Central procurement enables pooling of commitments, better discount tiers, and avoidance of overlapping or underutilized reservations. Individual teams provide demand forecasts quarterly to inform purchasing decisions.
