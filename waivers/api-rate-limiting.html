Rationale

The internal analytics API serving demand forecasting data to downstream consumers was deployed without rate limiting. The API currently serves 4 internal consumers with predictable traffic patterns. Adding rate limiting requires deploying a Redis cluster for distributed rate tracking, which was deferred to reduce initial deployment complexity.

Risk Description

Without rate limiting, a single misbehaving consumer can exhaust API capacity, causing denial of service for all downstream systems. The analytics API backs real-time inventory decisions, making availability critical during peak hours.
